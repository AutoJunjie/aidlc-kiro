# XHolacracy MVP - 术语表 (Glossary)

## 概述
本术语表定义了 XHolacracy 系统中使用的所有关键术语，包括 Holacracy 管理实践相关术语、系统组件术语和技术术语。所有需求文档中的术语必须与此术语表保持一致。

---

## Holacracy 管理实践术语

### 组织 (Organization)
整个组织实体，是所有圈子、角色和伙伴的最高层级容器。

**特性**:
- 包含所有圈子和角色
- 定义组织的整体目标和使命
- 是 Holacracy 治理结构的根节点

### 伙伴 (Partner)
填充角色的组织成员。一个伙伴可以同时填充多个角色，代表组织中的实际人员。

**同义词**: 角色填充者 (Role Filler)、成员 (Member)

**相关术语**: 角色分配 (Role Assignment)

### 角色 (Role)
组织中的一个功能单元，定义了特定的目的、职责和领域权限。一个角色可以由一个或多个人填充，一个人也可以填充多个角色。

**相关术语**: 角色填充者 (Role Filler)、角色分配 (Role Assignment)

### 角色填充者 (Role Filler)
被分配到特定角色的个人或用户。一个用户可以同时填充多个角色。

**同义词**: 角色承担者

### 目的 (Purpose)
角色存在的原因和目标，描述该角色要实现的长期目标或使命。

**示例**: "确保产品质量符合客户期望"

### 职责 (Accountability)
角色需要持续执行或维护的具体活动或输出。职责是可衡量和可追踪的。

**示例**: "每周发布产品质量报告"、"审核所有代码提交"

### 领域权限 (Domain)
角色拥有独占控制权的资源、资产或活动范围。其他角色需要获得许可才能影响该领域。

**示例**: "产品代码库"、"客户数据库"、"营销预算"

### 圈子 (Circle)
一组相关角色的集合，形成组织结构的层级单元。圈子本身也是一个角色，可以包含子圈子。

**特性**: 
- 具有层级结构
- 可以嵌套
- 拥有圈子引导者

### 子圈子 (Sub-Circle)
嵌套在父圈子内部的圈子，继承父圈子的某些属性和约束。

**相关术语**: 父圈子 (Parent Circle)

### 圈子引导者 (Circle Lead / 圈引导)
负责引导和协调圈子运作的特殊角色，负责圈的总体目标和资源分配。

**职责**: 
- 分配圈子内的角色给合适的伙伴
- 设定圈子优先级和策略
- 解决资源冲突和分配资源
- 当角色无人填充时自动填补该角色

**同义词**: Lead Link（引导连接）

### 协调员 (Facilitator)
主持治理过程和运营实践的特殊角色，确保会议和流程遵循 Holacracy 规则。

**职责**:
- 主持治理会议和战术会议
- 引导集成决策流程
- 监控圈子是否遵守宪法规则
- 验证反对意见是否满足有效标准
- 协助圈子成员解决反对和集成解决方案

### 秘书 (Secretary)
维护圈的记录和会议的特殊角色，负责文档管理和流程解释。

**职责**:
- 维护圈子的治理记录
- 安排圈子的治理和战术会议
- 解释宪法和治理记录中的规则
- 检查提案是否符合宪法要求
- 判定违反宪法的治理是否无效

### 圈代表 (Circle Rep)
将圈的需求在更广泛的圈（上级圈子）中表达的特殊角色。

**职责**:
- 将圈子内的紧张带入上级圈子
- 代表圈子在上级圈子中表达需求
- 参与上级圈子的治理决策
- 将上级圈子的决策和信息带回本圈子

**同义词**: Rep Link（代表连接）

### 政策 (Policy)
对圈子内角色行为的约束或规则，限制领域权限的使用方式或定义特定流程。

**示例**: "所有代码提交必须经过代码审查"、"预算支出超过 10000 元需要审批"

### 紧张 (Tension)
识别现状与期望之间的差距，是推动组织变革和改进的核心概念。任何伙伴都可以感知和提出紧张。

**特性**:
- 可能涉及流程瓶颈、缺乏资源、协作障碍等
- 是提案的来源和动力
- 可以在战术会议或治理会议中处理

**示例**: "客户反馈响应时间过长"、"角色职责不清晰导致工作重复"

### 连接角色 (Link Role)
连接两个圈子的特殊角色，用于在圈子之间传递信息和协调工作。

**类型**:
- **代表连接 (Rep Link / 圈代表)**: 从子圈子到父圈子的连接
- **引导连接 (Lead Link / 圈引导)**: 从父圈子到子圈子的连接
- **跨连接 (Cross Link)**: 连接两个非父子关系的圈子

### 提案 (Proposal)
解决紧张的建议变更，对组织结构、角色、政策或流程的修改建议。提案需要经过审批流程才能生效。

**来源**: 由伙伴基于感知到的紧张提出

**内容要求**:
- 紧张描述：说明当前问题
- 实际例子：提供具体场景
- 解决逻辑：说明提案如何解决紧张

**生命周期**: 创建 → 审批中 → 已批准/已拒绝 → 已应用/已关闭

### 提案类型 (Proposal Type)
提案的分类，不同类型的提案可能有不同的审批流程和阈值要求。

**包括**:
- 角色修改提案
- 政策调整提案
- 圈子结构变更提案
- 流程优化提案

### 治理流程 (Governance Process)
改变圈结构的正式流程，用于修改角色、政策和圈子结构。通常在治理会议中进行。

**核心流程**: 集成决策流程 (Integrative Decision-Making Process)

**步骤**:
1. 提案阶段：提出紧张和解决方案
2. 澄清问题：理解提案内容
3. 反应轮：表达看法和感受
4. 修改与澄清：提案人修改提案
5. 反对轮：表达反对意见
6. 集成阶段：解决反对，修改提案

**目的**: 确保组织结构能够持续进化和适应

### 治理会议 (Governance Meeting)
修改圈子的结构、角色、政策和选举的正式会议。

**频率**: 通常每月一次，或根据需要召开

**参与者**: 圈子成员 (Circle Members)

**流程**:
1. 签到轮：分享当前状态
2. 议程构建：提出治理事项
3. 处理议程项目：使用集成决策流程处理提案
4. 结束轮：分享会议感受

### 战术会议 (Tactical Meeting)
处理运营事务的定期会议，解决运营问题和日常协作事项。

**频率**: 通常每周或每两周一次

**参与者**: 圈子内所有角色负责人

**流程**:
1. 签到轮：分享当前状态
2. 清单审查：验证常规行动完成情况
3. 指标回顾：分享关键指标
4. 进展更新：分享项目进展
5. 议程构建：提出处理事项
6. 事项处理：明确行动、分配任务、协调活动
7. 结束轮：分享会议感受

### 审批流程 (Approval Process)
提案从创建到应用所经历的审批步骤和规则。

**特性**:
- 可配置的审批级别
- 支持并行或串行审批
- 基于阈值的自动化决策

### 阈值 (Threshold)
触发提案自动批准或应用的条件，通常基于投票结果或时间限制。

**类型**:
- 赞成票比例阈值
- 赞成票数量阈值
- 反对票限制
- 时间期限
- 特定角色批准要求

### 反对 (Objection)
对提案可能造成伤害的担忧，是集成决策流程中的关键环节。有效的反对必须满足特定标准。

**有效反对标准**:
- 会减少圈子实现目标的能力
- 会限制反对者履行职责的能力
- 不采纳提案时问题不存在
- 提案会造成伤害或错失适应机会

**处理方式**: 通过集成阶段寻找解决方案，修改提案以解决反对

**相关术语**: 反对轮 (Objection Round)、集成阶段 (Integration Phase)

### 投票 (Vote)
审批者对提案表达的意见，通常包括赞成、反对或弃权。

**相关术语**: 赞成票 (Approval Vote)、反对票 (Objection Vote)

## 概念关系图

本节描述 Holacracy 核心概念之间的关系和交互模式。

### 1. 组织层次结构

```
Organization（组织）
    ↓ 包含
Anchor Circle（锚圈）
    ↓ 包含
Sub-Circles（子圈子）
    ↓ 包含
Roles（角色）
    ↓ 可包含
Sub-Roles（子角色）
```

**关系说明**:
- Organization 是最高层级容器，包含整个组织的所有元素
- Anchor Circle 是组织的根圈子，直接隶属于 Organization
- 每个 Circle 可以包含多个 Sub-Circle，形成层级结构
- 每个 Circle 包含多个 Role
- 每个 Role 可以进一步分解为 Sub-Role

### 2. 特殊角色关系

每个 Circle 必须包含以下特殊角色：

```
Circle（圈子）
    ├── Circle Lead（圈引导）：管理 Circle 的整体运营和资源分配
    ├── Facilitator（协调员）：主持会议和流程，确保遵循规则
    ├── Secretary（秘书）：维护记录和解释宪法
    └── Circle Rep（圈代表）：代表 Circle 向上级圈子反映需求
```

**职责分工**:
- **Circle Lead**: 战略层面 - 设定方向、分配资源、分配角色
- **Facilitator**: 流程层面 - 引导会议、守护流程、验证反对
- **Secretary**: 文档层面 - 记录维护、规则解释、合规检查
- **Circle Rep**: 沟通层面 - 向上传递紧张、参与上级决策

### 3. 权限分配关系

```
Organization（组织）
    ↓ 授权
Anchor Circle（锚圈）
    ↓ 分配权限
Circle（圈子）
    ↓ 分配权限
Role（角色）
    ↓ 填充
Partner（伙伴）获得权限
```

**权限流动**:
- Organization 将整体权限授权给 Anchor Circle
- Circle 通过 Circle Lead 将权限分配给内部的 Role
- Partner 通过填充 Role 获得该角色的权限
- 权限包括：目的 (Purpose)、职责 (Accountability)、领域 (Domain)

### 4. 流程关系

#### 治理变更流程

```
Tension（紧张）
    ↓ 触发
Proposal（提案）
    ↓ 进入
Governance Process（治理流程）
    ├── Governance Meeting（治理会议）
    └── Integrative Decision-Making（集成决策）
        ├── 提案阶段
        ├── 澄清问题
        ├── 反应轮
        ├── 修改与澄清
        ├── 反对轮（Objection）
        └── 集成阶段
    ↓ 产生
Policy / Role Change（政策或角色变更）
```

#### 运营协作流程

```
Tension（紧张）
    ↓ 提出
Tactical Meeting（战术会议）
    ├── 签到轮
    ├── 清单审查
    ├── 指标回顾
    ├── 进展更新
    ├── 议程构建
    ├── 事项处理
    └── 结束轮
    ↓ 产生
Actions / Projects（行动或项目）
```

**流程区分**:
- **Governance Meeting**: 处理结构变更（角色、政策、圈子结构）
- **Tactical Meeting**: 处理运营事务（日常协作、项目推进、问题解决）

### 5. 决策机制

#### 基于同意而非共识

```
传统共识决策：需要所有人同意
    vs
Holacracy 同意决策：没有有效反对即可通过
```

**核心原则**:
- 不追求完美方案，追求"足够好且安全尝试"
- 反对不是个人偏好，而是基于伤害预测
- 通过快速迭代而非一次性完美来改进

#### 反对处理机制

```
Objection（反对）
    ↓ 验证
是否满足有效反对标准？
    ├── 是 → 进入集成阶段
    │        ↓
    │    修改提案以解决反对
    │        ↓
    │    返回反对轮
    │
    └── 否 → 视为无效反对，继续流程
```

**有效反对标准**:
1. 会减少圈子实现目标的能力
2. 会限制反对者履行职责的能力
3. 不采纳提案时问题不存在
4. 提案会造成伤害或错失适应机会

#### 集成决策流程 (Integrative Process)

```
提案 → 澄清 → 反应 → 修改 → 反对 → 集成 → 通过
         ↑                              ↓
         └──────── 如有新反对，循环 ────────┘
```

**关键特点**:
- 每个阶段有明确规则和时间限制
- 不允许跨阶段讨论
- 提案人保持提案所有权
- 必须解决所有有效反对
- 修改后的提案必须仍能解决原始紧张

### 6. 角色与伙伴关系

```
Partner（伙伴）
    ↓ 可填充多个
Role 1, Role 2, Role 3, ...
    ↓ 每个角色定义
Purpose（目的）+ Accountabilities（职责）+ Domains（领域）
```

**多角色模式**:
- 一个 Partner 可以同时填充多个 Role
- 一个 Role 可以由多个 Partner 填充
- Partner 在不同 Role 中拥有不同的权限和职责
- Partner 需要明确区分自己当前以哪个 Role 身份行动
